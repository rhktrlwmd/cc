/ Информация о проекте

CAFECORE.

Цель проекта

Создать единую, удобную и интуитивно понятную систему, которая автоматизирует процессы продаж и обеспечивает точный учет остатков сырья и ингредиентов в режиме реального времени.

Задачи разработки

  • 1. Разработка модуля продаж (POS): Реализовать быстрый интерфейс для оформления заказов, применения скидок и приема различных типов оплаты.
  • 2. Создание системы учета запасов (Inventory): Внедрить функционал для списания ингредиентов по факту продажи готового продукта (расход по рецептуре).
  • 3. Управление рецептурами: Разработать интерфейс для создания, редактирования и привязки состава готовых продуктов к списку сырья.
  • 4. Система оповещений: Настроить автоматические уведомления менеджера о критически низких остатках ингредиентов.
  • 5. Разработка отчетности: Реализовать отчеты по продажам (топ-продукты, средний чек) и списанию сырья за период.

Уровень Аудитория Роль в системе

Вторичный Повара/Кухня Получение уведомлений о новых заказах, отметка о готовности.
Периферийный Менеджеры/Владельцы Управление запасами, ввод новых ингредиентов, просмотр финансовой отчетности и аналитики.

Маркетинг-микс (4Р)

  • 1. Product(Продукт) Единое облачное веб-приложение, доступное на стационарных POS-терминалах и мобильных устройствах (для инвентаризации).
  • · Функция: POS-интерфейс, модуль списания по рецептам, модуль поставщиков.
  • 2. Price(Цена) Модель SaaS (Subscription as a Service), основанная на ежемесячной подписке.
  • · Тарифы: "Базовый" (только продажи), "Профи" (продажи + инвентаризация и отчеты), "Корпоративный" (для сети кафе).
  • 3. Place(Место распространения)
  • · POS-терминалы: Установка на рабочие места продавцов.
  • · Облачный доступ: Веб-интерфейс, доступный через любой браузер для менеджеров.
  • · Мобильное приложение: Доступ к модулю инвентаризации для сотрудников склада.
  • 4. Promotion(Продвижение) Акцент на экономию и эффективность:
  • · Слоган: "Зарабатывайте больше, контролируйте всё."
  • · Преимущества: Сокращение потерь от порчи продуктов, снижение трудозатрат на ручной учет, устранение ошибок бариста при приеме заказов.

Техническое задание

+

1. Планирование UML диаграмм

На этапе проектирования системы управления запасами и продажами кофе создается структура, позволяющая отслеживать движение товара (кофе в зернах, расходники) от склада до чашки клиента. UML диаграммы помогают визуализировать процесс списания остатков при каждой продаже.


2. Реализация логики системы

Реализация включает автоматизацию ключевых процессов:
  • 1. Продажа: Выбор напитка → Списание ингредиентов со склада по тех. карте → Печать чека.
  • 2. Пополнение запасов: Приемка кофе от поставщиков и обновление инвентаризационной ведомости.
  • 3. Аналитики:Уведомление о критически низком уровне запасов (например, когда осталось менее 2 кг зерна.

3. В жизни (эксплуатация системы)

Система используется в кофейнях для быстрого обслуживания гостей. Обеспечивается синхронизация между точками продаж и центральным складом, интеграция с платежными терминалами и ведение истории закупочных цен для контроля маржинальности.

Техническое задание

1. Общие сведения

Система предназначена для комплексной автоматизации операционной деятельности кафе. Основные цели: эффективный учет продаж (POS) и запасов сырья (Inventory Management) для минимизации потерь и ускорения обслуживания.

2. Требования к функционалу

2.1. Модуль Продаж (POS)

Быстрое создание заказа, автоматический расчет суммы, применение скидок, поддержка наличной и безналичной оплаты. Генерация уведомлений повару после оплаты.

2.2. Управление запасами

Автоматическое списание сырья при продаже по рецептурам (техкартам). Учет приходов, инвентаризация и корректировка остатков.

2.3. Аналитика и отчеты

Формирование отчетов по выручке, среднему чеку, топ-продуктам. ABC-анализ продаж и мониторинг расхода сырья в реальном времени.

3. Технические требования

3.1. Технологический стек

Backend: Node.js / REST API. Database: PostgreSQL. Frontend: Web-интерфейс (адаптивный дизайн). Deployment: Cloud-инфраструктура.

3.2. Надежность и безопасность

Сохранность данных при обрывах связи, ежедневное резервное копирование БД. Ограничение доступа к складским операциям для рядового персонала.

4. Пользовательские характеристики

4.1. Интерфейс (UI/UX)

Минимальное количество кликов для оформления заказа. Крупные элементы управления для сенсорных POS-терминалов. Темная тема оформления.

4.2. Система ролей

Кассир: только продажи. Менеджер: полный доступ к закупкам, ценам и отчетам. Повар: просмотр очереди заказов.

5. Программа тестирования

6. Критерии приемки и оценка качества

  • Реализация всех функциональных модулей (POS, Склад, Отчеты).
  • Отсутствие критических ошибок при имитации "часа пик".
  • Корректная работа системы автооповещений о низких остатках.
  • Наличие документации пользователя и схемы базы данных.

Реализация

1. Математические модели

1.1. Сети Петри (Расширение)

Назначение: Моделирование последовательности и параллельности бизнес-процессов (прием заказа, приготовление, списание).

  • · Применение: Идеально для моделирования логики автоматического списания сырья со склада по рецептуре.
  • · Элементы: Позиции (состояния: ЗаказПринят, ИнгредиентыНаСкладе), Переходы (события: Оплата, Списание_сырья), Цветные Маркеры (токены: сам Заказ с данными о составе).

    • 1.2. Системы массового обслуживания (СМО)

    Назначение: Анализ эффективности обслуживания клиентов и загрузки рабочих станций.

  • · Применение: Оценка времени ожидания в очереди и расчет оптимального количества персонала (бариста, повар).
  • · Элементы: Источники (Клиенты), Заявки (Заказы), Обслуживающие аппараты (Бариста, Повар), Очередь (Заказы, ожидающие приготовления), Параметры (интенсивность поступления \lambda, время обслуживания \mu).

    • 1.3. Модели системной динамики

    Назначение: Долгосрочный анализ и прогнозирование агрегированных процессов.

  • · Применение: Прогнозирование объемов закупок (RES), анализ влияния продаж (PR) на истощение запасов с течением времени.
  • · Элементы: Уровни (Накопители: УровеньЗапасаКофе, НакопленнаяВыручка), Потоки (Скорости: ПотокСписанияПоПродажам), Операторы (Op: управляющие функции, запускающие закупку).

    • 2. Динамическая модель процесса и представление знаний (ПЗ)


    2.1. Динамическая модель процесса списания запасов

    Модель описывает логику автоматического списания ингредиентов (RES) при завершении заказа.

    ЭлементРоль в системеПример действия/состояния
    Процесс (PR)PR.ЗакрытиеТранзакцииВесь цикл обработки оплаченного заказа.
    Операторы (Op)Op.ПроверитьРецептуруФункция, извлекающая из базы данных точные нормы расхода (количества) для каждого ингредиента.
    Агенты (Agent)Agent.СлужбаСписанияВнутренний системный модуль, выполняющий команду списания (V).
    Ресурсы (RES)RES.ИнгредиентАктуальный остаток сырья на складе (напр., RES.Молоко, RES.Кофе).
    Команды управления (V)V.УменьшитьОстатокКоманда на запись новых данных об остатках в БД.
    Источники (Sender)Sender.POS-МодульОтправляет сигнал "ЗаказОплачен" и P.СоставЗаказа.
    Приемники (Receiver)Receiver.СкладскойУчетМодуль БД, который принимает и обрабатывает V.УменьшитьОстаток.
    Параметры (P)P.КоличествоПроданоПараметр, определяющий, сколько единиц продукта было продано.
    Перекрестки (Junction)J.ПроверкаНаЛимитУсловие: Остаток RES > 0? (для предотвращения отрицательных остатков).

    Краткая логика:Когда Sender.POS-Модуль посылает сигнал "ЗаказОплачен", Op.ПроверитьРецептуру рассчитывает потребность. Если J.ПроверкаНаЛимит истинна, Agent.СлужбаСписания выполняет V.УменьшитьОстаток и передает команду Receiver.СкладскойУчет.
